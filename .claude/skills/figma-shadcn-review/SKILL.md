# Figma → shadcn Review

## Overview

This skill reviews code produced by `figma-implement-design` (or any Figma-to-code workflow) and replaces raw HTML elements with the project's installed shadcn/UI components. It reads the Figma design to understand visual intent, then applies the substitutions that best match both the design and the project's component API.

## When to Use

- After running `figma-implement-design` and the output contains `<button>`, `<select>`, `<input>`, ad-hoc card divs, or badge spans instead of components.
- When you notice stat cards, table action buttons, filters, or pagination built with raw HTML + custom CSS instead of the design system.
- To reduce dead/duplicate CSS and standardise components across the codebase.

## Skill Boundaries

- This skill only modifies code in the project repository. It does not write to Figma.
- It does not redesign or change visual behaviour — only substitutes equivalent components.
- It does not install new shadcn components. It only uses what is already installed.
- If a shadcn equivalent is not installed and the raw HTML implementation is adequate, leave it and document the gap.

## Prerequisites

- Figma MCP server must be accessible.
- The user must provide the Figma URL for the implemented design (or it must be known from context).
- The target file(s) to review must be identified.

## Required Workflow

**Follow these steps in order. Do not skip steps.**

### Step 1: Discover Installed shadcn Components

Before touching any code, scan the project to find which components are available.

```
Glob: UI/**/*.jsx  (or wherever the project keeps its design system)
Glob: components/ui/**/*.jsx
```

For each file found, note:
- **Component name and export** (e.g. `Button`, `Select`, `Card`, `Badge`)
- **Available variants and sizes** (read the `cva()` call or the component props)
- **Import path** relative to the target file

Build a substitution table before writing any code. Example:

| Raw HTML | shadcn Component | Variant/Size | Import path |
|---|---|---|---|
| `<button className="kl-table-action">` | `<Button>` | `variant="outline" size="sm"` | `../../UI/button` |
| `<select className="kl-filter">` | `<Select>` | — | `../../UI/select` |
| `<div className="kl-stat-card">` | `<Card>` + `<CardContent>` | — | `../../UI/card` |

### Step 2: Fetch the Figma Design

Run `get_screenshot` (and `get_design_context` for component names if needed) on the original node to understand the visual intent.

```
get_screenshot(fileKey=":fileKey", nodeId=":nodeId")
```

Use the screenshot as the source of truth for:
- Button styles (filled, outline, ghost, icon-only)
- Select/dropdown appearance
- Badge variants (colors indicate semantic states)
- Card structure (padding, label/value layout)
- Pagination style (outlined numbers, prev/next text buttons)

If `get_design_context` returns component names (e.g. `"Buttons/Button"`, `"Badge"`, `"_Button base"`), map them to shadcn variants using the table in Step 1.

### Step 3: Audit the Target File(s)

Read the target file(s) and produce a substitution list. For each raw element, determine:

1. **Is there an installed shadcn component that semantically matches?** (Button for interactive controls, Select for dropdowns, Card for container cards, Badge for status labels)
2. **Which variant/size matches the Figma visual?**
   - Filled/dark → `variant="default"`
   - Bordered, white background → `variant="outline"`
   - No border, no background → `variant="ghost"`
   - Icon-only square → `size="icon"`
   - Small secondary action → `size="sm"`
3. **Will the substitution break the layout?** (e.g. `Select` has `w-full` by default — may need `className="w-auto"`)
4. **Does the raw CSS class become dead after substitution?** Mark it for removal.

**Do not substitute when:**
- The component needs a visual state (multi-variant badge, active underline tab) that the installed component cannot express without breaking existing consumers.
- The substitution would require adding props to a shared component that is already stable.
- The raw element is inside a third-party component or a component you are not modifying.

### Step 4: Apply Substitutions

For each item in the substitution list:

1. Add the import at the top of the file (deduplicate if already imported).
2. Replace the raw element with the shadcn component.


### Step 5: Validate Against Figma Screenshot

Compare the updated component visually against the screenshot from Step 2.

Validation checklist:
- [ ] Buttons match their Figma variant (outline/filled/ghost/icon)
- [ ] Selects/dropdowns render correctly (check `w-auto` on flex containers)
- [ ] Stat cards maintain label + large value layout
- [ ] Pagination prev/next and number buttons are distinguishable
- [ ] Badge semantic colors are preserved (green = answered, red = error, etc.)
- [ ] No layout regressions from component size changes (Button `size="sm"` vs custom padding)
- [ ] Active tab state is visually correct (not overridden by ghost hover)
